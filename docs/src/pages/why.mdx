# Why?

## x402-Native, Not Chain-Native

Most wallet tooling is designed around the broader on-chain application world. Libraries like Wagmi are excellent when
an app needs chain-specific integrations, contract reads and writes, token balances, account abstraction flows, and DeFi
protocol composition.

Crosshatch starts from a different assumption: the product is not trying to expose the chain. The product is trying to
pay for a resource.

An x402-native wallet stack can hide the operational details that do not matter to the payment experience. The developer
works with payment requirements and payment payloads. The user approves an app-scoped allowance. Crosshatch handles the
wallet, signing, payload generation, and repeated execution inside that allowance.

## Stablecoin Rails Without Blockchain UX

Most developers and users do not want to reason about addresses, gas, chain IDs, contract methods, nonce management, or
account abstraction policy.

Crosshatch keeps the product surface close to the thing being bought:

:::steps

### Merchant returns a payment requirement

The merchant describes the resource price in x402 terms. The application receives a payment requirement.

### Client asks Crosshatch to propose payment

The client passes that requirement to Crosshatch, which handles the wallet work needed to turn the requirement into a
signed payment payload.

### The user approves an allowance when needed

On first run, Crosshatch asks the user to approve an app-scoped allowance. Future payments inside that boundary can
proceed without turning every resource request into a wallet prompt.

### The app receives signed x402 payloads

The application receives payment payloads it can send over the same transport as the resource request. It does not
receive private keys, custody funds, or need to manage gas, nonces, or contract calls directly.

### The merchant verifies payment and serves the resource

The merchant validates the payload against the original requirement, settles the payment according to its strategy, and
returns the resource. The product interaction stays centered on access to the resource, not on blockchain mechanics.

:::

The payment rail remains available, but the blockchain vocabulary does not leak into every application boundary.

## Offloading Operating Costs

x402 lets developers push payment requirements upstream to the consumer wallet. That matters when the app itself calls
paid merchants or incurs variable operating costs.

Instead of absorbing every downstream API call, model invocation, generated asset, retrieval, or compute-heavy operation
as an application cost, a developer can call an x402 merchant and yield the merchant's payment requirement to the user
wallet. The user pays for the resource they are consuming, and the application does not need to pre-fund every possible
execution path.

This is especially important for AI products. Token usage, tool calls, media generation, retrieval, simulation, and
agentic workflows can have unpredictable cost profiles. A fixed subscription, prepaid API key, or developer-funded quota
often maps poorly to real usage. x402 gives software a way to price work at the point where the cost is actually known.

## No API Key Registration

Traditional paid APIs require developers to register accounts, create API keys, manage billing relationships, and
provision credentials before software can call them. That model is a poor fit for generated applications, user-authored
automations, and AI systems that discover tools at runtime.

With Crosshatch, a call can encounter a `402 Payment Required`, request approval from the user wallet, and continue
without the developer pre-registering an account with every merchant. The payment relationship is formed at runtime
around the resource being purchased.

Generative UI can embed Crosshatch calls directly. Agents can discover paid tools. Applications can route payment
requirements back to the user's wallet instead of asking the developer to provision another billing credential.

## Realtime Payment Payloads

Crosshatch is designed for repeated x402 payload generation, not occasional wallet transactions. Once the user has
approved an allowance, an application can generate many payment payloads quickly without re-opening a wallet prompt for
every unit of work.

That unlocks interaction patterns that are awkward or heavily convoluted with account-abstraction-first wallet
approaches:

- realtime multiplayer payments
- pay-per-message or pay-per-action systems
- streaming media or generated assets priced by chunk
- agent workflows that pay for many small tool calls
- collaborative experiences where usage changes second by second

In these flows, the wallet is not just authorizing a transaction. It is continuously producing x402 payment payloads
inside a user-approved boundary.

## Holistic, Minimal Surface

Crosshatch intentionally avoids exposing every wallet feature. The goal is not to be the most flexible on-chain SDK. The
goal is to make x402 payments simple enough to embed anywhere.

The surface is small:

- interpret payment requirements
- ask the user for approval when needed
- manage app-scoped allowances
- generate signed payment payloads
- work across transports
- correlate payments with runtime traces
- keep key material out of application code

That is enough to make stablecoin payments programmable without forcing every developer to become a wallet engineer.

## Non-Custodial By Design

Crosshatch is non-custodial. Key material never leaves the user's device in decrypted form. The user controls the
wallet, and applications receive signed payment payloads rather than custody over funds or raw signing material.

This keeps the payment UX lightweight without turning Crosshatch into a custodian. Applications can request payment
capability, but they do not own the user's keys.

## The Short Version

Crosshatch is an x402-native wallet stack for paid software. It replaces account setup, API keys, generic wallet
plumbing, and chain-specific integration work with a focused payment surface: receive a requirement, approve an
allowance, generate payloads, and pay for resources as they are consumed.
