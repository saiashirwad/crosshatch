# OpenTelemetry

OpenTelemetry lets merchants connect payment activity to the traces that already describe resource requests, background
work, and settlement calls.

## Why Trace Payments

Payments are part of the request lifecycle. When a paid resource fails, the useful debugging question is rarely just
"did payment fail?" It is usually where the request moved through pricing, requirement creation, payload verification,
facilitation, settlement, and resource delivery.

Tracing those steps together makes it possible to answer questions like:

- Which request produced this payment requirement?
- Which payment payload paid for this resource?
- Did verification fail before or after facilitator interaction?
- Was the resource served after payment verification succeeded?
- Which settlement strategy was used for this purchase?

## Trace Shape

A typical paid request should be traceable as one connected flow.

::::steps

### Receive The Request

The merchant receives a request for a paid resource.

### Price The Resource

The merchant decides which x402 payment requirement applies.

### Emit A Requirement

If payment is missing, the merchant returns `402 Payment Required` with the requirement.

### Verify The Payload

The client retries with a payment payload. The merchant or facilitator verifies it.

### Settle Or Record Usage

The system settles immediately, records granular usage, or updates a deposit + rebate session.

### Serve The Resource

The merchant returns the paid resource.
