# Payload

`Payload` is Crosshatch's utilities for working with x402 payment payloads.

If `Required` is the invoice format of x402, `Payload` is the signed check. It
is what a payer creates after reading a payment-required invoice and deciding
how to pay it. The server or facilitator can then verify that signed check
against the selected requirement and settle the payment.

Every payload has a stable x402 envelope that says which invoice entry was
accepted, plus an inner payment body that is chain-specific. For EIP-155
payments, for example, the inner body may contain an EIP-3009 transfer
authorization or a Permit2 authorization plus signature.

## Making a Payload

Use `Payload.make` when you have a `Required` invoice and want to construct the
signed payment payload for it.

```ts
import { Config, Effect } from "effect"
import { Payload, Required, Requirements, KnownAssets } from "crosshatch"
import { Eip155Address } from "crosshatch/Eip155"

const program = Effect.gen(function* () {
  const required = yield* Required.make`Access to the paid resource.`.pipe(
    Required.accept(
      Requirements.asset(KnownAssets.USDC, {
        amount: 0.01,
        recipients: {
          eip155: {
            8453: yield* Config.schema(
              Eip155Address.Eip155Address,
              "PAY_TO_EIP155",
            ),
          },
        },
      }),
    ),
  )

  const { payload } = yield* Payload.make({ required })

  return payload
})
```

`Payload.make({ required })` returns an Effect because creating a payment
payload may need to select a supported requirement, read wallet configuration,
sign typed data, and encode chain-specific payload fields.

## The `Payer` Requirement

`Payload.make` does not know which wallet, account, chain, or
requirement-selection policy your application wants to use. Instead, calling it
introduces a `Payer` service requirement into the Effect.

```ts
import { Payer } from "crosshatch"

const { createPayload } = yield * Payer.Payer
```

At runtime, the provided `Payer` decides which `required.accepts` entry to pay
and how to produce the signed payload. For the built-in EIP-155 flow, the
`Payer` uses the configured asset metadata to select a supported requirement,
then asks the selected adapter to create the concrete payload.

Provide a `Payer` layer when running the program. See [Payer Service](/payer)
for local signing and bridge-backed payer setup.

To delegate payload creation to an external wallet, browser extension, or other
agent, provide `Payer.layerBridge` with a `Bridge` implementation instead of a
local signing chain.
