# Extensions

The extension system lets merchants ask clients for additional data to submit
alongside the means of payment.

Extensions are useful when the merchant needs context that is specific to a
payment requirement but does not belong in the chain-level transfer
authorization itself. For example, a merchant might request an idempotency key,
a client trace identifier, a fulfillment hint, an account reference, or another
structured value that helps the merchant process the paid request safely.

In this way, the merchant can call into the client through the requirement. The
requirement does not only say "pay this amount"; it can also say "include this
extra data when you pay." The client then returns the requested extension data
in the payment payload, together with the signed payment authorization.

## Defining An Extension

Extensions are typed Effect services. The merchant publishes an `info` value in
the `Required` envelope, and the payer produces an `enrichment` value in the
`Payload` envelope.

```ts
import { Extension } from "crosshatch"
import { Schema as S } from "effect"

export class ClientReference extends Extension.Service<ClientReference>()(
  "example/ClientReference",
  {
    identifier: "client-reference",
    info: S.Struct({
      required: S.tag(true),
    }),
    enrichment: S.Struct({
      required: S.tag(true),
      id: S.String,
    }),
  },
) {}
```

## Merchant API

Merchants request extension data with `Required.extend` and decode returned
extension data by installing the extension in the HTTP middleware.

```ts
import { ChxHttp, Required } from "crosshatch"
import { PaymentId } from "crosshatch/Extensions"
import { Effect } from "effect"

const program = Effect.gen(function* () {
  const required = yield* Required.make`Access to the paid resource.`.pipe(
    Required.extend(PaymentId.FromMerchant, {
      required: true,
      id: PaymentId.random(),
    }),
  )

  const middleware = ChxHttp.layerMiddleware({
    extensions: [PaymentId.FromMerchant],
  })

  return { required, middleware }
})
```

Inside a protected route, `yield* PaymentId.FromMerchant` returns the decoded
enrichment from the submitted payload, or `undefined` when the payload did not
include it. Handle a required value explicitly:

```ts
import { PaymentId } from "crosshatch/Extensions"
import { Effect } from "effect"

const program = Effect.gen(function* () {
  return yield* PaymentId.FromMerchant.pipe(
    Effect.flatMap(Effect.fromNullishOr),
  )
})
```

## Client API

Clients answer extension requests by registering handlers. A handler receives
the decoded `info`, selected requirement, generated scheme payload, and complete
invoice. It returns the typed enrichment embedded in the payment payload.

```ts
import { Extension } from "crosshatch"
import { PaymentId } from "crosshatch/Extensions"
import { Effect } from "effect"

const paymentIdHandler = Extension.layerHandler(
  PaymentId.FromClient,
  Effect.fn(function* ({ info, accepted, payload, required }) {
    return { required: info.required, id: PaymentId.random() }
  }),
)
```

Provide the handler alongside `Payer.layer`. When `Payer.layer` creates a
payload, it reads `required.extensions`, decodes any extension requests that
have registered handlers, and stores each handler's enrichment in
`payload.extensions`.

Provide the handler as part of your payer layer, alongside asset selection and
the signing schemes your client supports.

Extensions also expose explicit decoders for lower-level integrations:
`PaymentId.FromMerchant.decodeRequired(required)` reads the merchant's request
from a `Required`, and `PaymentId.FromMerchant.decodePayload(payload)` reads the
payer's decoded enrichment from a `Payload`.

For non-HTTP integrations, provide a decoded extension service directly from a
payload:

```ts
import { Extension, Payload } from "crosshatch"
import { PaymentId } from "crosshatch/Extensions"

declare const payload: Payload.Payload

const ExtensionLive = Extension.layerPayload(PaymentId.FromMerchant, payload)
```

At the wire-envelope level, extension maps are JSON records keyed by extension
identifier. Each extension's `info` and `enrichment` schemas supply type safety
when creating requirements, invoking handlers, and decoding payloads.

## Example: Payment Identifiers

Payment-specific identifiers are especially important because paid resources
often need safe retry behavior.

A merchant may include a payment-required-specific idempotency key so that
repeated submissions of the same payload can be recognized as the same purchase
attempt. This is useful when the client retries after a network interruption,
when the merchant needs to correlate verification with fulfillment, or when
settlement completes before the original transport response reaches the client.

The merchant may also request that the client provide its own identifier.
Client-provided identifiers are useful for observability because the client can
correlate payment creation, payload submission, facilitator verification, and
resource delivery across its own systems. They are also useful for retrieving
previously paid resources: the client can return with the same identifier and
the merchant can look up the earlier paid exchange instead of requiring a new
payment.

Crosshatch ships three variants of the built-in `payment-identifier` extension:

- `PaymentId.FromMerchant`: the merchant supplies the id in the requirement and
  the payer includes the same id in its enrichment.
- `PaymentId.FromClient`: the merchant requests a client-generated id. The
  payload enrichment may omit it.
- `PaymentId.FromEither`: the merchant may supply an id, but the final payload
  must contain one.
