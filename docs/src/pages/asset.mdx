# Assets

Crosshatch uses two closely related asset concepts: `Asset` and `PhysicalAsset`.

## `Asset`

`Asset` is the chain-specific asset identifier. On an EIP-155 chain, this is
typically the token contract address. It answers the narrow question: "which
onchain asset identifier is this?"

## `PhysicalAsset`

`PhysicalAsset` describes one logical asset across one or more concrete network
deployments. It groups the deployment metadata Crosshatch needs to build and
validate payments on each chain.

## Why "Physical?"

An asset is rarely just one identifier. The same logical asset, such as USDC,
can exist as different token contracts on different networks. Each deployment
has its own chain id, address, decimals, name, symbol, version, and
chain-specific payment implementation.

For EIP-155 assets, the network namespace is `eip155`, and each namespace's
references map to an ERC-20 deployment. This lets application code talk about a
single asset, while Crosshatch still preserves the exact physical deployment
needed for Base, Polygon, Arbitrum, or any other supported network.

## Known Assets

Crosshatch ships known asset definitions. These are ready-to-use `PhysicalAsset`
values for common tokens.

```ts
import { KnownAssets } from "crosshatch"

const { MEGAUSDC, MUSD, SBC, USDC, USDCE, USDT0 } = KnownAssets
```

Each export is a `PhysicalAsset`. For example, `USDC` contains multiple EIP-155
deployments under the `eip155` namespace:

```ts
import { KnownAssets } from "crosshatch"

const baseUsdc = KnownAssets.USDC.deployments.eip155[8453]
const polygonUsdc = KnownAssets.USDC.deployments.eip155[137]
```

Those two values refer to different physical token contracts, but they are
grouped under the same logical Crosshatch asset definition.

Known assets are useful when your application accepts a standard token and wants
Crosshatch to supply the deployment details. They also serve as examples for
custom asset definitions.

## Custom Assets

To represent your own asset, define the logical asset once, then add one entry
per supported deployment.

```ts
import { Eip155Asset, Erc3009, Permit2 } from "crosshatch/Eip155"
import type { Asset } from "crosshatch"

export const MY_TOKEN = {
  peg: "USD",
  symbol: "MYT",
  deployments: {
    eip155: {
      8453: {
        asset: Eip155Asset.Eip155Asset.make(
          "0x0000000000000000000000000000000000000000",
        ),
        decimals: 18,
        name: "My Token",
        version: "1",
        adapters: [Erc3009.Erc3009Adapter, Permit2.Permit2Adapter],
      },
    },
  },
} as const satisfies Asset.PhysicalAsset
```

Use a separate `PhysicalAsset` when deployments should not be treated as the
same logical asset. Native USDC and bridged USDC.e are a good example: they may
share similar names and denominations, but they have different issuance,
backing, and bridge-risk properties. Modeling them separately keeps those
differences visible to payment code.
