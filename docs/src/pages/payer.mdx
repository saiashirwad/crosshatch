# Payer

`Payer` is the service that turns a `Required` value into a signed x402
`Payload`.

Local clients usually provide it with `Payer.layer`, asset metadata for
requirement selection, and the supported signing schemes:

```ts
import { Accept, KnownAssets, Mnemonic, Payer } from "crosshatch"
import { Eip155Signer, Erc3009Scheme, Permit2Scheme } from "crosshatch/Eip155"
import { Layer } from "effect"

const PayerLive = Payer.layer.pipe(
  Layer.provide(
    Accept.layer(KnownAssets.Usd).pipe(
      Layer.provide(
        Layer.mergeAll(Erc3009Scheme.layer, Permit2Scheme.layer).pipe(
          Layer.provide(
            Eip155Signer.layerMnemonic.pipe(Layer.provide(Mnemonic.layerEnv)),
          ),
        ),
      ),
    ),
  ),
)
```

## Requirement Selection

`Accept` is the requirement-selection service used by `Payer.layer`.

Given a `Required` invoice, it chooses one compatible entry from
`required.accepts`, finds the matching physical asset, and returns the scheme
effect that can create the chain-specific payment payload.

`Accept.layer(denomination)` receives a denomination config. `KnownAssets.Usd`
is the built-in USD denomination Crosshatch ships. Provide it to `Payer.layer`
together with the signing schemes your client supports.

`Accept` compares the invoice's accepted payment options against the configured
denomination. A payment option is usable when its network and asset match a
configured physical asset, and one of that asset's schemes is available in the
current Effect context.

Most applications do not call `Accept` directly. They provide `Accept.layer` to
`Payer.layer`, then call `Payer.createPayload`.

If no configured asset and scheme can satisfy the invoice, payment creation
fails. This usually means the client does not support the merchant's requested
asset, network, or payment scheme.

## Extensions

The built-in payer also answers registered client extensions. For every
recognized entry in `required.extensions`, it decodes the extension's `info`
schema and invokes the registered `Extension.layerHandler` with
`{ info, accepted, payload, required }`. The returned enrichment is encoded with
the extension's `enrichment` schema and stored in `payload.extensions`.

Unknown extensions and extension values that fail schema decoding are skipped.

## Bridge Payer

If payment creation should be delegated to another agent, provide
`Payer.layerBridge` with a `Bridge` implementation. That keeps HTTP and
`Payload.make` integrations Payer-based while allowing signing and user approval
to live elsewhere.
